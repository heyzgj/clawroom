owner's stated boundary."** This is the failure the rest of this plan

## 3. Refinements before implementation

### 3A. Do NOT add `waiting_owner` state to the relay

The temptation is to track per-role wait state on the DO so the
counterpart bridge knows "the other side is paused, don't spam." This
reintroduces server-side semantic state — the same trap v2 fell into.

The thin solution: **the bridge that's waiting just stops calling its
LLM and stops posting.** The other bridge keeps long-polling and sees no
new messages. Long-poll is already the synchronization primitive — no
relay change needed. The same-role 409 turn gate already prevents the
counterpart from posting twice in a row.

If the counterpart's LLM eventually says "you've been quiet, want me to
move things along?" — that's a perfectly fine message to post. It's not
spam. It's how the bridge recovers from "owner taking too long."

**Rule:** the relay learns about ASK_OWNER only as another message kind
(see 3B). It does not learn about waiting state.

### 3B. Owner-reply primary surface = Telegram reply, URL is fallback

The owner is in Telegram. The bridge's notification reaches them in
Telegram. The most natural reply path is: owner replies to the
notification message in Telegram → bot's webhook receives the reply →
bot routes it to the bridge.

Designing a separate URL endpoint as the PRIMARY owner-reply path
forces the owner to leave Telegram and visit a webform. UX hostile,
adds a credential surface, and contradicts the "ClawRoom is to-agent /
through Telegram" memory.

**Recommended owner-reply paths (in priority order):**

1. **Telegram reply to the notification message** — primary. Bot's
   inbound handler must recognize messages whose `reply_to_message_id`
   matches a known ASK_OWNER notification it sent on behalf of the
   bridge, and route them to the bridge's relay endpoint instead of
   the main agent.
2. **Signed URL with one-time token** — fallback for cases where
   Telegram reply isn't possible (owner on web, owner away from
   Telegram). Same token model as 3C.

The relay endpoint underneath them is the same — but the bot does the
routing, not the user.

### 3C. Token model for owner-reply

Whatever the surface, the relay endpoint that accepts the reply must:

- Use a token DIFFERENT from `host_token` and `guest_token`. Call it
  `owner_reply_token`.
- Be **scoped to one specific question**: minted by the relay when the
  bridge posts an ASK_OWNER, returned to the bridge in the post
  response, embedded in the notification.
- Be **single-use**: relay marks it consumed on first successful reply.
- Have **TTL** (recommend 30 min). After expiry, owner's reply is
  rejected; bridge can re-ask if needed.
- Carry a **question_id** the relay records. Owner-reply MUST cite the
  question_id; replies to a stale question are rejected.

This avoids: replay, cross-question contamination, indefinite-validity
tokens leaking via Telegram screenshots.

### 3D. Lesson F2 (notification-as-instruction) mitigation in design

The bridge's Telegram notification will land in the same chat where the
owner normally talks to their bot. Even though we use direct Bot API
(bypassing OpenClaw `deliver`), the bot's own webhook handler will see
the owner's reply.

Two design rules to land before T3:

1. **Notification text must be unambiguously a system message, not user-
   sounding text.** Recommended template:
   ```
   🔒 ClawRoom · Authorization needed
   Room: t_xxxxxxx-xxx
   Topic: <topic>
   Question: <agent's ASK_OWNER text>

   Reply to this message:
   • "yes" / "approve" → authorize as proposed
   • "no" → reject; agent will continue within ceiling
   • free-form text → forwarded as your decision
   ```
   This makes any reasonable bot's main session ignore it (no
   instruction verb, system framing, contains room id which the main
   session shouldn't act on).
2. **Bot's inbound handler explicitly intercepts replies whose
   `reply_to_message_id` is in the known-ASK_OWNER set** and routes to
   the bridge owner-reply endpoint. **Does not pass them to the main
   agent.** This is the single most important mitigation — even if
   notification text leaks, the routing layer blocks contamination.

### 3E. Multi-question is per-role, not per-thread

Both host and guest can independently hit ASK_OWNER. The relay's
ASK_OWNER state (and `owner_reply_token`) must be keyed by `(thread_id,
role, question_id)`, not just by `thread_id`. Don't assume "only one
side waits at a time."

### 3F. Timeout policy

- ASK_OWNER notification sent at T₀.
- T₀ + 15 min: bridge sends a Telegram reminder ping ("still waiting
  on your reply for room t_xxx").
- T₀ + 60 min: bridge can either auto-cancel the room or escalate. For
  T3 E2E, default: auto-cancel with stop_reason `owner_timeout`. The
  validator will check this is handled cleanly (counterpart notified,
  both runtimes stopped, no orphan poll).
- Configurable per room via launch prompt parameter, but ship a sane
  default first.

### 3G. Mandate parser in bridge as defense-in-depth (the most important refinement)

This is the **hybrid** that codex didn't propose but I think is essential.

LLM-driven ASK_OWNER (LLM decides when to ask) is fragile. The same
LLM that we're trying to gate WILL forget to ask sometimes — that's
literally the failure mode that produced `t_0b3602a9-e3b`.

**Defense-in-depth: bridge parses the host owner_context for explicit
mandate constraints, AND independently checks every CLOSE summary
against them. If a CLOSE attempts to terminate the room with terms that
violate the mandate, the bridge intercepts:**

1. Block the CLOSE from being posted to the relay.
2. Synthesize an internal ASK_OWNER citing the violation.
3. Notify owner via Telegram.
4. Resume only after authorized owner reply (which can either approve
   the violation or instruct agent to re-negotiate).

For this to work, the host owner_context needs SOMETHING parseable.
Two acceptable forms:

- **Free-form with marker convention**: lines like
  `MANDATE: budget_ceiling = 65000` or
  `MANDATE: refuse_observer_seat`. Bridge parses `^MANDATE:\s*(.+)$`
  and tries simple shape match (`<key> = <value>`,
  `refuse_<keyword>`, etc.).
- **Explicit JSON block** at the end of owner_context:
  ```
  --- mandate ---
  {"budget_ceiling_jpy": 65000, "refuse": ["observer_seat"]}
  ```
  Bridge parses the JSON.

Recommend supporting both. The MANDATE: line is friendlier for the
owner (still natural-ish), the JSON block is robust for programmatic
authoring (e.g., via a future "ClawRoom open this room" UI).

**This is NOT putting structure back in the relay.** The relay still
sees only opaque message text. The mandate parser lives entirely in
`bridge.mjs`. The owner-context format is a contract between owner and
their own bridge — not part of the inter-agent protocol.

### 3H. Lesson AI applies to ASK_OWNER marker too

When you add ASK_OWNER marker recognition, EXTEND the same tolerant
parser:

```
/^\s*(REPLY|CLAWROOM[_ ]CLOSE|ASK[_ ]OWNER)\s*[:：]/i
```

Plus: when the parser sees an unmatched turn whose text contains
keywords suggesting an authorization request — `owner`, `approval`,
`permission`, `authorize`, `授权`, `批准`, `请示`, `老板`, `boss`,
`确认` — it should LOG the turn loudly and treat it as a soft
ASK_OWNER candidate. Don't auto-route it (false positives possible),
but do flag it in the heartbeat / artifact so the validator and the
owner can spot prompt drift.

## 4. Recommended sequence (differs from codex's order)

Codex's plan was: marker → endpoint → state → E2E → validator.

Mine flips it: **validator first**, so we have a regression test for
the failure we just observed before we attempt the fix.

### Step 1 (smallest, fastest, do first) — Validator additions

Add to `scripts/validate_e2e_artifact.mjs`:

- `mandate_compliance` check. Inputs: artifact's redacted owner_context
  for each role (need to start including this in the redacted artifact —
  add a `owner_contexts` field with parsed mandates only, NOT the full
  free text), plus the close summary. Logic:
  - If a mandate exists for the host (e.g., `budget_ceiling: 65000`)
    AND the close summary names a value above ceiling AND no
    `OWNER_REPLY` event in the transcript explicitly approved the
    excess → `fail`.
  - If no mandate exists → `n/a` (don't fail).
  - If mandate exists, close within bounds → `pass`.
- `ask_owner_evidence` check. If the run reached close and the mandate
  was binding, validator records whether at least one `OWNER_REPLY`
  appears in the transcript. Required for T3 pass.

**Run validator on `t_0b3602a9-e3b` after this change.** Expected
result: was green, becomes RED on `mandate_compliance`. That's the
regression test confirmation. Commit the validator change with a
sentence in the commit body that says "this would have caught
t_0b3602a9-e3b."

(If you can't add `owner_contexts` to past artifacts because they were
generated before the field existed, that's fine — the validator can
report `n/a` for past artifacts and `pass`/`fail` for new ones. The
point is the gate exists for the future.)

### Step 2 — Owner-reply protocol on paper (15-30 min, no code)

Write a one-pager `docs/protocol/owner-reply.md` (new dir; this is a
durable spec, not a progress artifact) with:

- Endpoint shape: `POST /threads/:id/owner-reply`
- Request body schema: `{ token, question_id, text, role }`
- Token rules (Section 3C above)
- Multi-question semantics (Section 3E)
- Timeout policy (Section 3F)
- Telegram-reply routing rules (Section 3D)
- Failure modes: stale token, wrong question_id, expired, replay,
  malformed → all return specific 4xx codes the bridge can reason about

This doc becomes the contract. After it lands, Step 3 just implements
the contract.

### Step 3 — Minimal owner-reply endpoint on relay

Add to `relay/worker.ts`:

- New endpoint per the spec.
- DO state additions: per-role `pending_question` field with
  `{question_id, owner_reply_token, posted_at, ttl_at}`. When set,
  posts from that role's bridge that aren't OWNER_REPLY don't get
  rejected by the relay (relay stays semantic-free) — instead the
  BRIDGE knows it's waiting and doesn't post. The DO field is just
  bookkeeping for the validator and for owner-reply token verification.
- New event kind `owner_reply` with the redacted reply text and the
  question_id it answered. Appears in the transcript so the validator
  can find it.

**The new endpoint is the only protocol surface added.** No new GET
endpoints, no new opaque state.

### Step 4 — Bridge: ASK_OWNER + mandate parser + waiting state + Telegram notify

In `bridge.mjs`:

- Marker parser extended (Section 3H).
- New state: `waiting_owner` (per role, in runtime-state, with
  `question_id` and `notified_at`).
- When `ASK_OWNER:` is parsed: post an `ask_owner` event to relay (no
  text relay event added — keep mailbox shape; or add an
  `ask_owner_marker` event kind, design call), receive
  `{question_id, owner_reply_token}`, send Telegram notification per
  3D template, enter `waiting_owner`.
- While in `waiting_owner`: continue heartbeat, do NOT call OpenClaw,
  do NOT post messages. Periodically check relay for an `owner_reply`
  event matching `question_id`.
- On `owner_reply` event observed: clear `waiting_owner`, inject
  reply text into the next OpenClaw call's prompt as
  `OWNER_REPLY: <text>`, resume.
- Mandate parser (Section 3G): on every CLOSE the LLM tries to emit,
  pre-flight check the close summary against parsed mandate. On
  violation, intercept → synthesize ASK_OWNER → enter waiting_owner.
- Telegram inbound handler: must intercept replies to known ASK_OWNER
  notification message IDs and POST to `/threads/:id/owner-reply`.
  Out-of-scope replies pass through to main session as before.


## 5. Anti-patterns to actively avoid

- **Don't make the relay parse mandates.** Mandate is owner-bridge
  contract. Relay sees opaque messages.
- **Don't add a "waiting" state to relay-side same-role 409 logic.**
  The 409 is mechanical — it doesn't need to know about wait. Bridge
  knows.
- **Don't make the `--min-messages` style gate the answer to mandate
  compliance.** That worked for "negotiate at least N rounds" because
  the constraint is countable and content-free. Mandate compliance
  needs semantic understanding of values vs ceiling — that's the
  bridge's mandate parser, not a flag.
- **Don't trust LLM compliance with ASK_OWNER alone.** The whole point
  of this layer is that the LLM did not voluntarily ASK_OWNER in
  `t_0b3602a9-e3b`. Defense-in-depth (bridge mandate parser) is what
  actually closes the gap. ASK_OWNER from LLM is the polite path; the
  bridge intercept is the safety net.
- **Don't ship T3 without the bot inbound handler routing rule
  (Section 3D rule 2).** Without it, owner reply text becomes a new
  instruction to the main agent and the demo will fail in confusing
  ways. This is Lesson F2 manifesting in v3.

## 6. What "T3 passed" means (binary criteria)

A T3 run is a pass if and only if ALL of:

1. Validator green on every check (existing 9 + 3 new).
2. The transcript contains at least one ASK_OWNER → owner_reply →
   continued negotiation cycle.
3. The close summary respects the mandate as documented in
   owner_context, OR explicitly cites an owner approval that was
   itself recorded as an OWNER_REPLY.
4. The defense-in-depth bridge intercept fires at least once across
   the three-run T3 series (run the same scenario 3 times to see
   variance — sometimes LLM will ASK voluntarily, sometimes the
   bridge will need to force it).
5. Bot main session does NOT respond to the ASK_OWNER notification
   text as if it were a new instruction. Verifiable via main session
   conversation logs in OpenClaw state dir.

Failing run: commit as
`docs/progress/v3_1_<room_id>.failed.redacted.json` per the artifact
discipline. Failure is data.

## 7. Open product question (worth surfacing, not a blocker)

The `--min-messages` gate is fine for "must negotiate at least N
rounds" because that's a countable structural constraint. But: how
many of these will we accumulate? `--min-messages`, `--max-rounds`,
`--require-summary-keyword`, `--ban-jargon`, `--max-cost`?

If we end up with 8 of these, we're back to v2 thick-protocol territory
in a different costume. Worth keeping a soft count and revisiting if
the list grows past ~3.

For now: 1 (`--min-messages`) is fine. Add the mandate parser. If we
need a 3rd kind of gate before the 5th run, revisit the structure.
